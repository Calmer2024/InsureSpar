# InsureSpar 保险销售对练系统 - 接口与业务技术文档

## 1. 核心架构与底层机制
InsureSpar 采用了 **LangGraph 状态机** 作为系统的流转核心。

### 1.1 业务状态定义 (`DialogueStage`)
系统定义了 6 种标准状态，任何对话必然在下面这几个状态中流转：
- `INTRODUCTION` (破冰与探寻)：开局必须的状态，主要了解客户情况。
- `OBJECTION` (异议处理)：解答客户疑虑、讲解产品。
- `DECISION_PENDING` (同意核保)：客户初步被说服，同意走下一步。
- `DECISION_SIGN` (签单成功)：客户强烈表示要立即签约。
- `DECISION_FOLLOW_UP` (需要跟进)：客户需要时间考虑，择日再议。
- `DECISION_REJECT` (客户拒绝)：明确表示不需要、反感。

**核心业务防线**：
1. **强制引入过渡**：从第1轮直接开始记作 `INTRODUCTION`，节省一次大模型算力。
2. **五轮防线**：任何对话前 5 轮强制拦截在 `INTRODUCTION` 或 `OBJECTION`。禁止系统过早判断为交易成功，逼迫系统进行多轮交互。
3. **两轮决策观察期** (`DECISION_STRIKES_REQUIRED = 2`)：为了避免单据冲动判决（例如客户只是随口夸一句，就被判定签单），模型必须在“连续 2 轮”都判定到了决策状态（如：同意投保），这局对练才算真正的 `is_finished=True`。

### 1.2 Agent 职能
- **🎯 对话管理器 (DM)**：隐藏在客户节点之后，作为内部分析总控。每当客户说完话后，DM 根据“上一轮客户的回复”+“本轮销售的话术”，决断下一个业务阶段。
- **🧑‍💼 客户 Agent**：作为对手盘，带工具。能调用工具查询费率、检查健康告知。
- **👓 考官 Agent**：旁观者。利用后台线程异步运行。根据每一轮销售的话进行 【专业性、合规性、策略性】（0-10分）的打分和建议。
- **🧠 销售 Agent (限 Auto 模式)**：按照指定的策略，取代人类自动生成推销话术发起攻击。

## 2. API 接口模块规划
系统开放了 3 组大类的 API 路由（可以通过 `/docs` 查看 Swagger）：

### 2.1 人机手动对练 (Manual)
提供给真实销售人员使用的前端接口。

| 方法 | 路径 | 核心业务逻辑 |
|:---|:---|:---|
| **POST** | `/api/session/create` | 选中一个画像生成唯一 `session_id`。 |
| **POST** | `/api/chat/send` | (早期遗留同步接口)，销售发送一句话，卡顿等待客户思考完返回文字。 |
| **POST** | `/api/chat/stream` | **(推荐使用)** 基于 SSE 的多路流式推送接口。发送一句话后，会流式拿到工具调用过程、回复打字机特效以及防线拦截系统警告。 |
| **GET** | `/api/session/{id}/history`| 获取整局明文历史。 |
| **GET** | `/api/session/{id}/evaluation`| 获取考官打出的异步分数（可能比当前对话轮次慢一轮，属正常现象）。 |
| **GET** | `/api/personas` | 拉取可用客户画像清单。 |

### 2.2 全自动对战 (Auto)
通过指定画像和对战策略，自动执行“AI互搏”用于回放学习。

| 方法 | 路径 | 核心业务逻辑 |
|:---|:---|:---|
| **GET** | `/api/auto/strategies` | 获取系统内置的各种策略（如：咨询型、数据流型）。 |
| **POST** | `/api/auto/session/create`| 给定画像+策略，创建自动局。 |
| **POST** | `/api/auto/step` | **核心触发器(SSE)**：前端点一次推进一步，后台自动生成销售话术->客户思考反驳->状态判定，一路以流式推到前端。 |
| **GET** | `/api/auto/session/{id}/final-report`| 当结局为`is_finished=True`后调用。生成500字总监点评并归纳6维度雷达能力盘，**同时触发全量落盘持久化**。 |

### 2.3 历史记录与查询
一切落盘于 MySQL 的业务查询入口（无账号隔离体系，凭 `session_id` 找数据）。

| 方法 | 路径 | 核心业务逻辑 |
|:---|:---|:---|
| **GET** | `/api/history/sessions` | 分页拉取大局列表，用于在前端展现历史场次汇总。 |
| **GET** | `/api/history/sessions/{id}`| 穿透详情。返回全量的对局明细（四张数据库大表 JOIN 级拼合输出）：局信息 + 所有对话话术 + 所有详细打分 + 总监雷达报告。 |

## 3. MySQL 持久化表结构
（配置通过 `app.core.config.DATABASE_URL` 指定，由于 FastAPI 的特性通过 `SessionLocal` 操作）

1. **`sessions`（总表）**：记录开局/结局时间、画像策略、总轮数等。
2. **`conversation_logs`（对话表）**：1场对决对应N条明文（`role`(sales/customer) 与 `content`）。
3. **`evaluations`（评分表）**：记录考官 3 大维度的数字分数外加 4 段文字点评（包含了综合指导 advise）。
4. **`final_reports`（报告表）**：一对一关系。以 JSON 格式原样保留雷达图所需的数据切片，以及大段综合评语。

## 4. 给前端开发同学的交互与调用流程指南
本系统涉及大量的后端长期任务和打字机特效推流，强烈推荐前端严格按下面 3 个生命周期来进行联动加载：

### 阶段一：创建房间 (初始化阶段)
1. 页面加载完毕后，拉取 `/api/personas`（画像）与 `/api/auto/strategies`（如果要在自动对战大厅）。
2. 点击"开始生成局"，请求 `POST /api/auto/session/create` (或者人机模式的 `POST /api/session/create`) 。
3. 后端返回带有 `session_id` 的对象包，前端**必须**在全局状态中把 `session_id` 缓存下来。

### 阶段二：SSE 心跳式对抗流解析 (对战阶段)
由于对话是由 AI 运算，耗时极长（包含思考、切工具、查库、组织语言），前端 **绝对不能** 使用普通 Ajax 等待 JSON 返回。必须使用 原生 `EventSource` 或是 `fetch` 读取 stream 数据。

当发出进攻指令 `POST /api/auto/step` 落定后，逐行监听后端返回的如下事件进行不同 UI 的渲染。
**(参见第 5 节的 SSE 流式事件速查表)**

### 阶段三：对局结束，拉取终极打分 (复盘阶段)
在 SSE 流式事件中，如果收到了 `stage_update` 并且里面的 `is_finished=True`，代表系统已经打死结，对局结束！
此时前端应执行：
1. **禁用下方的对话输入框和“下一步”按钮**，打上“已生成最终结局”的遮罩。
2. 弹出按钮让用户去拉取总评审告。
3. 点击请求 `GET /api/auto/session/{session_id}/final-report`，由于总监模型(DeepSeek)要阅读几十句话并深度归纳，**这个接口通常需要 10～15 秒**！前端必须挂出很明显的 Loading Toast（“AI总监正在撰写长篇复盘……”）防止用户急躁退出。
4. 收到数据后，把 `review` 拿来打印，把 `radar` 丢给 Echarts / Chart.js 画图。

## 5. SSE 流式事件数据格式速查表
以下事件全部产生于 `POST /api/chat/stream` 以及 `POST /api/auto/step` 之中，以标准的 `data: {"type": "xxx", "content": "..."}` JSON 格式下发。

| `type` 字段 | 含义 | 前端建议处理方式 |
|:---|:---|:---|
| `phase` | 大节点状态通告符 (如：“🤖 销售Agent正在行动...”) | 使用灰色小字直接输出在屏幕中间的日志区。 |
| `sales_token` | **自动局专属**：销售AI正在吐字！ | 在右侧/绿色的销售气泡中，执行打字机追加逻辑。 |
| `status` | 客户大节点通报 (如：“客户正在回应...”) | 同样灰字追加进中间的日志区。 |
| `tool_call` | 客户 AI 决定调用工具 (如检查费率) | 渲染成一个带 loading 的框“🔄 正在核查费率表...”，增加科技感。 |
| `tool_result` | 客户 AI 使用完工具得到结果 | 渲染成带打勾符号的灰色字“✅ 获取完成：每年 xxxx 元”。 |
| `token` | **重头戏**：客户正在逐字反驳或同意。 | 左侧/白色气泡中，执行打字机追加特效，这是主要的聊天区域。 |
| `stage_update` | 对话管家 (DM) 已下发本轮评估判定！数据里含有 `stage_label`(当前中文环节), `is_finished`(是否死局), `turn_count`(总轮数) | **核心监听**。更新左上角的“当前环节”、“总回合数”。如果发现 `is_finished=True`，就直接全盘锁定游戏界面。 |
| `evaluation_done` | **异步**：后台的考官把某一轮的分已经打完了。 | (可选，前端如果有轮次详细分可以点个红点提醒用户该轮有指教了)。 |
| `error` | 服务端遇到代码或大模型请求问题。 | 弹窗报错，终止该局。 |
